Project, Connettori & Skill

Stamattina abbiamo visto come funziona il contesto. Ora lo mettiamo dentro il prodotto.

Il Progetto è il posto dove il contesto smette di essere una teoria e diventa qualcosa che riusi ogni giorno — e quando il lavoro vive in una cartella, quello stesso contesto prende la forma di un file: il CLAUDE.md. I connettori sono il modo in cui il contesto arriva dal mondo esterno. In mezzo c'è un protocollo, MCP, che vale la pena capire una volta per tutte. E quando il contesto che ti serve non l'hai scritto tu ma qualcun altro — un mestiere già impacchettato, pronto da importare — c'è un ultimo pezzo: le skill.

01 Chat, Cowork, Code

Un solo modello, tre prodotti

Claude, Claude Cowork e Claude Code sono tre prodotti agentici diversi. Ma il modello sotto è lo stesso: le stesse capacità, gli stessi pesi, la stessa finestra di contesto. Quello che cambia sono l'harness — l'impalcatura di tool, permessi e ciclo di lavoro che abbiamo visto stamattina — e l'interfaccia da cui ci arrivi.

02 Il Progetto

Il Progetto: il contesto che si riusa

Un Progetto è un contenitore di contesto persistente. Ogni chat che apri al suo interno parte già con dentro le stesse cose. Nell'interfaccia sono tre pannelli — Istruzioni, Memoria, Contesto — e ognuno dei tre corrisponde a un pezzo della context window che abbiamo smontato stamattina.

03 CLAUDE.md

CLAUDE.md: le stesse istruzioni, ma in un file

Le Istruzioni di un Progetto vivono dentro l'app. Ma quando l'agente lavora sui tuoi file — in Cowork, in Code — le istruzioni scendono con il lavoro: diventano un file di testo dentro la cartella, il CLAUDE.md, che l'agente rilegge all'avvio di ogni sessione. Stesso ruolo, stesso posto in finestra. Cambia che questo lo puoi versionare, passarlo a un collega e ritoccarlo mentre lavori.

Cosa ci si scrive dentro — due assi
CLAUDE.md — nella cartella del progetto

Un buon file di contesto copre entrambi gli assi. Il primo dice dove si trova l'agente; il secondo dice come deve comportarsi mentre ci sta dentro. Nel file stanno vicini, in testa vanno tenuti distinti.

Dove vive il file

Non ce n'è uno solo. Ci sono tre posti, e i file si sommano: dal più generale al più specifico.

01 Personale — vale ovunque

In ~/.claude/CLAUDE.md, sul tuo computer. Le tue preferenze di sempre: rispondi in italiano, niente entusiasmo, mostrami cosa cambi prima di cambiarlo. Vale in ogni progetto, e non lo vede nessun altro.

02 Di progetto — vale per chi apre la cartella

Nella radice della cartella di lavoro. Le regole della casa: cos'è questo progetto, chi sono gli attori, cosa non si tocca. Sta insieme al lavoro, quindi lo versioni e chi eredita il progetto eredita anche il contesto.

03 Di sottocartella — vale solo lì dentro

Un CLAUDE.md in una sottocartella viene letto quando l'agente ci entra a lavorare. È il posto delle eccezioni locali — «i contratti in questa cartella sono riservati: non citarli mai per esteso» — senza appesantire il file principale.

04 MCP

MCP: il server si presenta e accetta testo

MCP (Model Context Protocol) è il protocollo con cui l'harness parla con il mondo esterno. L'idea è più semplice di quanto suggerisca il nome: chi vuole offrire capacità a un modello espone un server; il server, alla prima domanda, dice a parole cosa sa fare; e da quel momento riceve richieste in testo e restituisce testo.

La conversazione tra harness e server
Client — l'harness
Claude, Cowork, Code: chi ospita il modello
System
Sei Claude. Rispondi in italiano.
Utente
Trova la mail di Rossi sul preventivo e salvala nella cartella Clienti.
Server MCP
Chi espone le capacità: Gmail, Drive, Calendar, Linear, GitHub…
In attesa. Il server non sa nulla di te finché non gli parli.
gmail__search_threads
Cerca conversazioni nella casella dell'utente. Usalo quando serve trovare una mail per mittente, oggetto o contenuto.
query: string · limit: number
drive__create_file
Crea un file su Google Drive in una cartella indicata. Usalo per salvare un documento o un allegato.
name: string · folder: string · content: string

Ogni strumento ha un nome, una descrizione in linguaggio naturale e i suoi argomenti. Nient'altro.

// il server esegue e risponde. In testo. Trovata 1 conversazione da: m.rossi@studio.it oggetto: "Preventivo Q3 — revisione" allegato: preventivo_q3.pdf
 
Cosa succede lato modello
// la risposta del server entra dritta nella context window { "tools": [ { "name": "gmail__search_threads", "description": "Cerca conversazioni nella casella dell'utente. Usalo quando serve trovare una mail per mittente, oggetto o contenuto." }, { "name": "drive__create_file", "description": "Crea un file su Google Drive in una cartella indicata…" } ] }

Il modello non «impara» il server: lo legge. Quelle descrizioni occupano token nella finestra, esattamente come il resto. Sono ciò su cui deciderà.

La chiamata che il modello produce
// il modello non esegue: propone. Con argomenti scritti in testo. { "tool": "gmail__search_threads", "arguments": { "query": "Rossi preventivo", "limit": 3 } }

È lo stesso schema di stamattina: il modello propone, l'applicazione esegue. MCP standardizza solo il formato della busta.

05 Connettori

Un connettore è un server MCP

La domanda arriva sempre: «ma allora i connettori sono un'altra cosa rispetto a MCP?». No. Un connettore è un server MCP impacchettato per essere usato da chiunque: ospitato da qualcuno, con il login già gestito, verificato, e installabile con un click da una directory. Il protocollo sotto è lo stesso che abbiamo appena visto.

Tre strati, uno sopra l'altro
01 Il protocollo — MCP

Le regole della conversazione: come un server dichiara i suoi strumenti, come si chiamano, che forma hanno le risposte. È uno standard aperto: non appartiene a un prodotto.

02 L'implementazione — il server

Qualcuno scrive un server che parla MCP e lo mette davanti a Gmail, a Drive, al gestionale della tua azienda. Da solo, però, è ancora roba da installare e da autenticare a mano.

03 Il packaging — il connettore

Lo stesso server, ospitato in remoto, con l'autenticazione (OAuth) già risolta, i permessi espliciti, la verifica di sicurezza e una voce in una directory. Lo attivi con un interruttore, in Claude, in Cowork o in Code.

 Server MCP «nudo»Connettore
Che cos'èUn programma che espone strumenti secondo il protocolloLo stesso programma, distribuito come prodotto
Chi lo installaChi sa configurarloChiunque, dalla directory, con un click
Chi lo ospitaTu, dove decidi tuChi lo pubblica, in remoto
AutenticazioneDa gestireLogin e permessi già integrati
Cosa vede il modelloIdentico nei due casi: un elenco di strumenti descritti a parole, e testo che va e viene

L'ultima riga è il punto della lezione. Sopra al protocollo cambia la comodità, non la sostanza.

Il connettore non è un'alternativa a MCP: è il modo in cui MCP arriva all'utente finale. Quando attivi Gmail su Claude non stai usando «un'integrazione Google»: stai autorizzando un server MCP a descriversi al modello e a ricevere richieste in testo.
06 Skill

La skill: contesto fatto da qualcun altro

Fermiamoci a guardare da dove è arrivato, finora, tutto il contesto. Il prompt lo scrivi tu. Le Istruzioni del Progetto le scrivi tu. Il CLAUDE.md lo scrivi tu. I connettori portano dentro dati dai tuoi software — ma le porte le apri tu. Una skill rompe lo schema: è contesto scritto da qualcun altro — istruzioni, materiali, piccoli script — impacchettato in una cartella che importi in Claude. E nota la differenza con quello che abbiamo appena visto: non collega nessun software esterno. Aggiunge una capacità.

Il connettore dà a Claude un accesso. La skill gli dà un mestiere.
Due modi diversi di estendere Claude
 MCP / ConnettoreSkill
Cosa aggiungeUn accesso: Claude raggiunge un software esterno — Gmail, Drive, il gestionaleUna capacità: Claude impara un modo di fare — le slide aziendali, il controllo di una nota spese
Cosa c'è dietroUn server che gira da qualche parte, davanti a un software vivoUna cartella: testo, materiali, script. Nessun software esterno
Cosa entra in finestraLe descrizioni degli strumenti, e i risultati delle chiamateKnow-how: istruzioni e procedure — contesto, della stessa pasta di quello che scrivi tu
Chi lo faChi possiede il softwareChiunque abbia il know-how — e tu lo importi

L'ultima riga è quella nuova. Fin qui il contesto era artigianato: lo scrivevi tu, per te. Con le skill diventa una cosa che circola: qualcuno la scrive una volta, la impacchetta, la usano tutti.

07 La domanda giusta

«Ma questo non potevo scriverlo nel CLAUDE.md?»

Obiezione legittima. Se una skill è contesto, e il CLAUDE.md è contesto, perché due strumenti? Il CLAUDE.md l'avete appena imparato: sembra il posto naturale per tutto. La differenza non è cosa ci scrivi — è quando lo paghi. Il CLAUDE.md entra in finestra a ogni sessione, tutto intero, sempre. Una skill no. E siccome la finestra è scarsa — l'abbiamo visto smontandola pezzo per pezzo — questa differenza decide tutto.

Lo stesso know-how, tre posti — guardalo in due chat diverse
Chat A — «Fammi le slide del lancio»
in finestra
Chat B — «Riassumimi questo contratto»
in finestra

Il know-how è identico nei tre casi: la procedura per le slide aziendali, ~2.000 token. Quello che cambia è chi la paga, e quando.

Serve una volta

Nel prompt

Lo scrivi, lo usi, sparisce. Nessuna infrastruttura per un'esigenza che non torna.

Serve sempre

Nel CLAUDE.md

Le regole della casa: chi sei, cosa non si tocca, come parlarmi. Poche righe, valide in ogni sessione.

Serve a volte

In una skill

Un mestiere intero, on demand. In finestra una riga; il resto solo quando lavora.

08 Anatomia

Dentro la cartella: istruzioni, materiali, script

Com'è fatto, questo contesto impacchettato? Una cartella con un file d'ingresso obbligatorio — SKILL.md — più i materiali che nessun prompt può sostituire e, quando serve, piccoli script eseguibili. Un documento di riferimento risponde a «cos'è X»; una skill risponde a «come si fa X, in che ordine, con quali controlli».

La documentazione dice che cos'è. Una skill dice come si fa.

09 L'ecosistema

Se è una cartella, si importa

Una skill è una cartella di testo: si zippa, si manda, si pubblica. Ed è qui che cambia la scala. Il CLAUDE.md è artigianato — lo scrivi tu, per te. La skill è un formato di distribuzione: il know-how di qualcun altro, già distillato, che installi in trenta secondi. Sapere dove trovarle vale quanto saperle scrivere.

01 Le skill di Anthropic — già installate

Su claude.ai le trovi in Personalizza → Skill: Excel, Word, PowerPoint, PDF. Se ti sei mai chiesto perché Claude produce un .xlsx ben formattato: non è il modello, è una skill. Quel know-how l'ha scritto qualcuno — e lo stai già usando senza saperlo.

02 Il repository ufficiale — github.com/anthropics/skills

Le skill di Anthropic in versione open source, più una collezione di esempi: da scaricare, leggere, adattare. È il catalogo da cui vengono le due skill di oggi — brand-guidelines e theme-factory — e dentro Claude Code lo trovate già apparecchiato: sotto Skill, elencate e pronte da installare. È anche il posto giusto per imparare come si scrive una skill fatta bene: copiando dagli originali.

03 Le directory di community — il marketplace

Cataloghi di skill pubblicate da altri utenti e aziende: liste curate su GitHub, directory dedicate — e in Claude Code interi marketplace di plugin che si installano con un comando. Qualità variabile, ed è per questo che prima di installare si legge (sotto).

Il valore dell'import: dentro una skill ben fatta ci sono ore di tentativi di qualcun altro — i formati che rompono i file Office, i casi limite di una nota spese, l'ordine giusto dei passaggi. Tu la importi in trenta secondi. È il modo più veloce che esista di dare a Claude una competenza che non hai — o che non hai il tempo di scrivere.

E se la skill che ti serve non esiste ancora?

Strada 1 · Conversazionale

Chiedila a Claude

«Creami una skill che…». Claude attiva skill-creator — una skill che costruisce skill: ti fa domande, scrive SKILL.md, prepara materiali e script, impacchetta tutto in uno ZIP.

Strada 2 · Manuale

Cartella → ZIP → upload

Scrivi i file a mano, comprimi con la cartella come radice (slide-aziendali.zip → slide-aziendali/ → SKILL.md) e carica da Personalizza → Skill. È la stessa strada con cui ritocchi una skill importata.

Prerequisito per gli script: in Impostazioni → Funzionalità deve essere attivo «Esecuzione di codice e creazione di file» — è la sandbox in cui girano. Le skill custom caricate su claude.ai restano private del tuo account.

Una skill è software che dai in mano all'agente — e nel momento in cui la importi da un estraneo, questo conta doppio. Può orientare quello che Claude fa ed esegue: installa solo da fonti di cui ti fidi, leggi SKILL.md e gli script prima di abilitarla, diffida di skill che scaricano contenuti da URL esterni, e non scriverci dentro segreti — API key, password — da nessuna parte.
10 Demo

Prima io: un pezzo con lo stile di qualcun altro

Adesso il ciclo dell'import lo faccio davanti a voi, dall'inizio alla fine: prendo una skill dal catalogo, la installo, e produco un'infografica in uno stile che non è il mio e non è di Claude — è quello di Anthropic, impacchettato nella skill brand-guidelines. Mentre lo faccio, guardate tre cose.

1 · Da dove viene

Il catalogo, dentro Claude Code

Sotto Skill ci sono le skill di Anthropic: pronte, ma non installate. brand-guidelines la installo da lì, senza uscire dal terminale.

2 · Cosa chiedo

Una richiesta normale

Chiedo l'infografica come la chiedereste voi: descrivo il contenuto. La skill non la nomino mai — si attiva da sola, perché la richiesta matcha la sua description.

3 · Da dove esce lo stile

Dalla cartella, non dal modello

Palette, font, regole tipografiche: stanno nei materiali della skill. Senza, Claude uno «stile Anthropic» se lo sarebbe inventato — plausibile, e sbagliato.

11 Hands on

Ora voi: importa, e cambia stile

Stesso giro che avete appena visto, altra skill dello stesso catalogo: theme-factory — dieci temi grafici professionali pronti da applicare a qualunque pagina. L'obiettivo non è la pagina: è toccare con mano quanto stile vi portate a casa senza averlo scritto.

Trovala nel catalogo

In Claude Code, sotto Skill: le skill di Anthropic sono elencate, pronte ma non installate. Individuate theme-factory e installatela da lì.

Guarda cosa hai importato

Aprite la cartella della skill: SKILL.md con la description, e dieci temi in themes/ — palette, font e gerarchie già decisi. È l'anatomia della sezione 08, dal vivo. Nota: qui la fonte è Anthropic; con una skill presa da un marketplace di terzi questa lettura si fa prima di installare.

Crea il pezzo

Chiedete un'infografica sullo stesso topic di quella che vi ho appena mostrato — richiesta scritta com'è naturale scriverla, senza nominare la skill. Aprite il file nel browser.

Varia il tema

Chiedete l'elenco dei temi disponibili e fatevi rifare la stessa pagina in due o tre temi diversi. Stesso contenuto, stile ogni volta coerente — perché Claude non lo sta improvvisando: lo sta leggendo dalla cartella.

La controprova

Disattivate la skill e rifate la richiesta descrivendo il tema a parole. Confrontate: il delta di coerenza — palette, tipografia, gerarchie — è esattamente il contesto che avevate importato.

12 Da portare a casa

Quattro idee da portare a casa

01

Il prodotto è l'harness

Chat, Cowork e Code condividono il modello. Quando scegli tra loro non scegli un'intelligenza, scegli gli strumenti e i permessi che le dai.

02

Il Progetto è contesto riusabile

Istruzioni, Contesto, Memoria: gli stessi blocchi della context window, con un'interfaccia sopra. Nella cartella lo stesso ruolo lo fa il CLAUDE.md. Scrivilo una volta, riusalo ogni giorno.

03

MCP è un accesso

Il server si descrive a parole e scambia testo; il connettore è quel server col login già fatto. Ti collega a un software esterno — ma non insegna niente a Claude.

04

La skill è contesto importato

Il know-how di qualcun altro, in una cartella: in finestra vive solo la description, il corpo entra quando serve. Una volta → prompt. Sempre → CLAUDE.md. A volte → skill.
