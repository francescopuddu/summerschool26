01 Chat, Cowork, Code

Un solo modello, tre prodotti

Claude, Claude Cowork e Claude Code sono tre prodotti agentici diversi. Ma il modello sotto è lo stesso: le stesse capacità, gli stessi pesi, la stessa finestra di contesto. Quello che cambia sono l'harness — l'impalcatura di tool, permessi e ciclo di lavoro che abbiamo visto stamattina — e l'interfaccia da cui ci arrivi.

Scegli il prodotto e guarda cosa cambia davvero
Modello l'LLM
Harness tool, permessi, loop
Interfaccia da dove ci parli

Il primo strato non cambia mai. Cambiano gli altri due — ed è lì che nasce il prodotto.

Stamattina abbiamo chiuso dicendo che tutto quello che sta intorno al modello è l'harness. Questa è la conseguenza pratica: harness + interfaccia = prodotto. Non stai scegliendo un'intelligenza diversa, stai scegliendo un ambiente diverso in cui farla lavorare.

Come scegliere, in pratica

Chat

Tu esegui

Ragioni, scrivi, analizzi. Claude produce testo, poi sei tu a portarlo dove serve. Nessun accesso ai tuoi file, se non quelli che carichi.

Cowork & Code

Esegue lui

Stesso motore agentico: pianifica, usa strumenti, verifica il proprio lavoro. Cowork lo fa su documenti e app, Code su una repository.

02 Il Progetto

Il Progetto: il contesto che si riusa

Un Progetto è un contenitore di contesto persistente. Ogni chat che apri al suo interno parte già con dentro le stesse cose. Nell'interfaccia sono tre pannelli — Istruzioni, Memoria, Contesto — e ognuno dei tre corrisponde a un pezzo della context window che abbiamo smontato stamattina.

La context window della prossima chat
Le Istruzioni di un Progetto sono l'equivalente, in interfaccia, del CLAUDE.md che avete visto stamattina: un file di testo che descrive come si lavora qui, riletto a ogni sessione. Stesso ruolo, stesso posto in finestra. Cambia solo che uno lo scrivi in un campo e l'altro lo committi in una repo.
La «memoria» non è ricordo: è testo che qualcuno rimette in finestra al posto tuo.
Il Progetto è il posto dove decidi quale testo.

Ed è per questo che il Progetto è la risposta più diretta alla domanda di stamattina — «come faccio a non ripetere ogni volta le stesse premesse?». Il contesto buono costa fatica a costruirlo: il Progetto è il posto dove lo costruisci una volta e lo riusi.

03 MCP

MCP: il server si presenta e accetta testo

MCP (Model Context Protocol) è il protocollo con cui l'harness parla con il mondo esterno. L'idea è più semplice di quanto suggerisca il nome: chi vuole offrire capacità a un modello espone un server; il server, alla prima domanda, dice a parole cosa sa fare; e da quel momento riceve richieste in testo e restituisce testo.

La conversazione tra harness e server
Client — l'harness
Claude, Cowork, Code: chi ospita il modello
System
Sei Claude. Rispondi in italiano.
Utente
Trova la mail di Rossi sul preventivo e salvala nella cartella Clienti.
Server MCP
Chi espone le capacità: Gmail, Drive, Calendar, Linear, GitHub…
In attesa. Il server non sa nulla di te finché non gli parli.
gmail__search_threads
Cerca conversazioni nella casella dell'utente. Usalo quando serve trovare una mail per mittente, oggetto o contenuto.
query: string · limit: number
drive__create_file
Crea un file su Google Drive in una cartella indicata. Usalo per salvare un documento o un allegato.
name: string · folder: string · content: string

Ogni strumento ha un nome, una descrizione in linguaggio naturale e i suoi argomenti. Nient'altro.

// il server esegue e risponde. In testo. Trovata 1 conversazione da: m.rossi@studio.it oggetto: "Preventivo Q3 — revisione" allegato: preventivo_q3.pdf
 
Cosa succede lato modello
// la risposta del server entra dritta nella context window { "tools": [ { "name": "gmail__search_threads", "description": "Cerca conversazioni nella casella dell'utente. Usalo quando serve trovare una mail per mittente, oggetto o contenuto." }, { "name": "drive__create_file", "description": "Crea un file su Google Drive in una cartella indicata…" } ] }

Il modello non «impara» il server: lo legge. Quelle descrizioni occupano token nella finestra, esattamente come il resto. Sono ciò su cui deciderà.

La chiamata che il modello produce
// il modello non esegue: propone. Con argomenti scritti in testo. { "tool": "gmail__search_threads", "arguments": { "query": "Rossi preventivo", "limit": 3 } }

È lo stesso schema di stamattina: il modello propone, l'applicazione esegue. MCP standardizza solo il formato della busta.

Il server si descrive

Non serve scrivere un'integrazione per ogni coppia modello-servizio. Il server dichiara i propri strumenti a parole, e qualunque modello capace di leggere può usarli.

L'interfaccia è testo

Richieste e risposte sono testo strutturato. Nessuna libreria da importare lato modello: se sai descrivere cosa fai, sei già integrabile.

Ne discende una conseguenza pratica che vi tornerà utile per tutta la settimana: la qualità di una descrizione è la qualità dell'integrazione. Un tool con una descrizione vaga verrà usato male, o non verrà usato affatto. La documentazione non accompagna l'interfaccia: è l'interfaccia.

04 Connettori

Un connettore è un server MCP

La domanda arriva sempre: «ma allora i connettori sono un'altra cosa rispetto a MCP?». No. Un connettore è un server MCP impacchettato per essere usato da chiunque: ospitato da qualcuno, con il login già gestito, verificato, e installabile con un click da una directory. Il protocollo sotto è lo stesso che abbiamo appena visto.

Tre strati, uno sopra l'altro
01 Il protocollo — MCP

Le regole della conversazione: come un server dichiara i suoi strumenti, come si chiamano, che forma hanno le risposte. È uno standard aperto: non appartiene a un prodotto.

02 L'implementazione — il server

Qualcuno scrive un server che parla MCP e lo mette davanti a Gmail, a Drive, al gestionale della tua azienda. Da solo, però, è ancora roba da installare e da autenticare a mano.

03 Il packaging — il connettore

Lo stesso server, ospitato in remoto, con l'autenticazione (OAuth) già risolta, i permessi espliciti, la verifica di sicurezza e una voce in una directory. Lo attivi con un interruttore, in Claude, in Cowork o in Code.

L'ultima riga è il punto della lezione. Sopra al protocollo cambia la comodità, non la sostanza.

Il connettore non è un'alternativa a MCP: è il modo in cui MCP arriva all'utente finale. Quando attivi Gmail su Claude non stai usando «un'integrazione Google»: stai autorizzando un server MCP a descriversi al modello e a ricevere richieste in testo.
05 Hands on

Adesso lo costruiamo

Un Progetto vero, con dentro un contesto vero e almeno un connettore attivo. L'obiettivo non è finire per primi: è vedere, alla fine, che cosa è finito in finestra e perché.

Crea il Progetto

Dagli il nome di un lavoro che fai davvero, non un titolo di prova. Il contesto ha senso solo se è il tuo.

Scrivi le Istruzioni

Chi sei, per chi lavori, come vuoi le risposte, cosa non deve mai fare. Poche righe, in italiano, imperative. È il tuo CLAUDE.md.

Carica il Contesto

Due o tre documenti di riferimento, non venti. Un contesto affollato è un contesto peggiore: l'abbiamo visto stamattina.

Attiva un connettore

Drive o Gmail. Guarda i permessi che ti chiede prima di dire di sì: quello che concedi, il modello lo può usare.

Fai una richiesta che li usi entrambi

Qualcosa che richieda insieme i tuoi documenti e il connettore. Esempio: «confronta questo preventivo con le condizioni standard nel documento di riferimento».

Guarda cosa è successo

Quali strumenti ha chiamato? Che testo gli è tornato indietro? Riesci a ricostruire la finestra che aveva davanti quando ha risposto?

Due avvertenze, e sono le stesse di sempre. Primo: ogni connettore attivo occupa contesto — le descrizioni dei suoi strumenti entrano in finestra a ogni chat, anche quando non servono. Tenerne dieci accesi «per sicurezza» peggiora le risposte. Secondo: un connettore agisce a tuo nome. Concedi i permessi che servono, non tutti quelli che puoi.
06 Da portare a casa

Tre idee da portare a casa

01

Il prodotto è l'harness

Chat, Cowork e Code condividono il modello. Quando scegli tra loro non scegli un'intelligenza, scegli gli strumenti e i permessi che le dai.

02

Il Progetto è contesto riusabile

Istruzioni, Contesto, Memoria: gli stessi blocchi di stamattina, con un'interfaccia sopra. Costruiscilo una volta, riusalo ogni giorno.

03

MCP è descrivere, non programmare

Il server dice a parole cosa sa fare e scambia testo. Il connettore è quel server con il login già fatto. Sotto, il protocollo è lo stesso.
