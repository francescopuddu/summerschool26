01 Git essenziale

Prima di tutto: i checkpoint del lavoro

Con Claude Code si lavora per iterazioni veloci: si prova, si cambia, a volte si sbaglia. Git è il modo per avere checkpoint del proprio lavoro e poter tornare indietro quando una vibata va male — e, come vedrete nel pomeriggio, è ciò che abilita Claude Code in cloud.

Versionare

Salvare istantanee del progetto man mano che lavori. Se qualcosa si rompe, torni all'istantanea di prima.

È il "salva partita" del codice.

GitHub

Il posto nel cloud dove le istantanee vengono conservate: backup del progetto e — dal pomeriggio — la porta d'ingresso di Claude Code in cloud.

Il punto di partenza è banale: il progetto è una cartella sul vostro computer — file dentro una directory, niente di più. Git le aggiunge due cose: una memoria (ogni commit salva una nuova versione della cartella) e una copia nel cloud (il push la sincronizza su GitHub). Due gesti soli: niente branch, niente mondo infinito di Git.

I tre gesti si fanno solo in quest'ordine: prova a premere gli altri due, sono bloccati finché non tocca a loro. Poi si ricomincia — le versioni si accumulano.

02 Perché un'app

Oggi non chiediamo un artifact: costruiamo un'app

Ieri con Cowork e gli artifact avete già costruito piccoli strumenti — la matrice, un singolo file HTML autocontenuto. Oggi l'obiettivo è un gestionale per la manutenzione dei treni: dati che devono restare e aggiornarsi, regole che girano da sole, un ERP da interrogare, l'AI dentro. Un singolo file non basta più.

Artifact

un file, un foglio — e finisce lì

Una mini-app in un file HTML: interfaccia, logica e dati vivono tutti lì dentro. Perfetto per la matrice di ieri.

Ma i dati non sopravvivono e non si condividono, non si aggancia l'ERP, non ci lavorano più persone. Il tetto si tocca in fretta.

App

dati che restano, servizi che girano, l'AI a chiamata

Un progetto vero: dati che restano, un'interfaccia per gli utenti, regole e calcoli che girano da soli, collegamenti ad altri sistemi, l'AI chiamata via API.

È la forma dei tool interni che da oggi potete costruirvi.

03 L'architettura, già decisa

Le scelte tecniche ve le diamo noi: in un file

Un'app è fatta di pezzi, e sceglierli — quale database, quale linguaggio, come si parlano — è l'unico punto dove servirebbe davvero un ingegnere. Quel lavoro è già fatto: un file di testo con le scelte architetturali, da mettere nella cartella del progetto. Claude Code lo legge da solo e ci si attiene. A voi resta solo la consegna.

Tre pezzi, ognuno col suo mestiere, che si parlano tramite API. Non dovete progettarli né saperli configurare: sono le scelte già scritte nel file qui sotto. Le quattro parole servono solo a seguire cosa succede dove.

📄 CLAUDE.md il file delle scelte architetturali — Claude Code lo legge da solo 
# CLAUDE.md — Gestionale manutenzione treni (Summer School)

Sei il builder di questo progetto. Le scelte di architettura sono già state fatte e **non sono in discussione**: qualunque funzionalità ti venga chiesta, va costruita dentro questi binari. Se una richiesta sembra richiedere di uscirne, fermati e spiega il problema invece di improvvisare.

## Le scelte (fisse)

| Pezzo | Scelta |
|---|---|
| Database | PostgreSQL 16 in Docker (`docker compose up -d`), porta 5432, dati in un volume |
| Back-end | Python 3.12 + FastAPI, avviato con uvicorn sulla porta 8000 |
| Front-end | React + Vite, JavaScript semplice (niente TypeScript), dev server sulla porta 5173 |
| API | REST in JSON tra front-end e back-end; il front-end parla **solo** col back-end |
| AI | chiamate al modello **solo dal back-end**, con la libreria `anthropic`; chiave e nome del modello letti da `.env` |

## Struttura del progetto

```
progetto/
├── docker-compose.yml      # solo il database
├── CLAUDE.md               # questo file
├── README.md               # i comandi per avviare tutto (tienilo aggiornato)
├── .env / .env.example     # configurazione e segreti (.env NON si committa)
├── backend/                # FastAPI: API, logica, accesso al DB, chiamate AI
└── frontend/               # React + Vite: pagine e componenti
```

## Regole

- **Semplicità prima di tutto.** Chi guida questo progetto non è uno sviluppatore. Niente autenticazione, niente TypeScript, niente ORM/framework/librerie oltre a quelli indicati — se qualcosa in più ti sembra davvero necessario, proponilo prima e spiega perché.
- **Database.** Schema creato da uno script rilanciabile (SQL o SQLAlchemy `create_all`) + script di seed. Niente sistemi di migrazione.
- **Segreti.** `ANTHROPIC_API_KEY` e ogni credenziale vivono solo in `.env`: mai committati, mai nel front-end. Tieni aggiornato `.env.example` con le variabili richieste.
- **Avvio in 3 comandi.** `docker compose up -d`, avvio del back-end, avvio del front-end. Il README li riporta sempre, aggiornati e copia-incollabili.
- **Git.** A ogni iterazione che funziona, proponi commit + push (messaggi brevi, in italiano).
- **Spiega mentre lavori.** Chi legge non conosce il codice: quando finisci una modifica, di' in una riga *dove* hai toccato — database, back-end o front-end.

Va messo, con il nome esatto CLAUDE.md, nella cartella del progetto prima del primo prompt: Claude Code lo legge automaticamente all'avvio, non serve nemmeno dirglielo.

Perché il file: senza vincoli, dieci gruppi otterrebbero dieci stack diversi — e problemi non confrontabili. Col file le app di tutta l'aula si somigliano, gli errori pure, e l'aiuto tra i banchi funziona. Non serve capire ogni riga: serve sapere che c'è — e che per i vostri progetti potete farvene scrivere uno simile da chi sa, o da Claude stesso.

05 Da vedere

Demo: la dashboard dei treni

Un gestionale per la manutenzione ordinaria dei mezzi, costruito dal vivo: dallo schema di prima all'app funzionante. Ogni passo usa il vocabolario appena introdotto.

Demo · dashboard treni

Il punto di partenza. Una cartella con dentro solo il CLAUDE.md dell'architettura. Il primo prompt è la consegna del gestionale: Claude legge il file e tira su i tre pezzi da solo — nessuna scelta tecnica da fare dal vivo.

La fonte dati: un ERP mockato. Un componente preparato in anticipo che, ricevute le chiamate, risponde con dati presi da un JSON prefatto — mezzi, manutentori con le loro disponibilità, revisioni con le scadenze. "Mockare" è a sua volta un concetto da fissare: per costruire l'app non serve il sistema vero.

La dashboard. Visione d'insieme dei mezzi con lo stato delle revisioni e dei controlli obbligatori a scadenza: otto mezzi di categoria tre, tre con revisione fatta, cinque da fare.

Il bottone "Organizza revisione". Crea l'agenda del giorno o della settimana dei manutentori, incrociando revisioni da fare e disponibilità.

Human in the loop. L'agenda proposta si rivede: solo quando l'utente conferma, parte la mail ai manutentori con il programma ("il tuo tragitto del giorno è questo").

L'AI dentro il software. La composizione della mail la scrive un modello, chiamato via API dall'app stessa. La mail allega anche la scheda treno del mezzo, la documentazione da portare con sé.

06 Da fare

Hands-on: il documento del revisore

Esercitazione su consegna assegnata. Non un'app nuova: una pagina in più della stessa app appena vista — così si riusa tutto il contesto della demo. E niente scelte tecniche neanche qui: il CLAUDE.md è già nella cartella, a voi serve solo descrivere bene la consegna.

Hands-on · ~2 ore

La consegna

Il revisore, dopo il giro sui mezzi, produce un documento destrutturato: appunti sparsi su treni visti e interventi da fare. Costruite la pagina in cui quel documento si carica e diventa una lista di interventi strutturata, collegata alle revisioni di quel revisore, pronta per chi li prenderà in carico.

Gli step

La pagina di caricamento. Una box in cui trascinare il documento del revisore (fornito già pronto insieme alla consegna).

Il collegamento. L'app aggancia il documento alle revisioni che quel revisore doveva fare — i dati sono già nel database della demo.

L'estrazione. Dal testo destrutturato l'app estrae la lista degli interventi: qui entra l'AI dentro il software, con la chiave API distribuita in aula.

Modifica e conferma. Gli interventi estratti si possono correggere e integrare; poi OK, e vengono salvati — qualcun altro li prenderà in carico.

Descrivere, non digitare

Con Claude Code il software si costruisce raccontandolo. Il vostro mestiere è la consegna, non la sintassi.

AI nel software

Non solo AI che scrive il codice: app che, una volta costruite, usano un modello per lavorare (la mail, l'estrazione).

Nel pomeriggio il codice esce dal vostro computer: Claude Code in cloud, deploy su Fly.io, e il database che migra su Supabase.
