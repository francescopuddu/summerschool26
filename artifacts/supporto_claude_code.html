01 Git essenziale

Prima di tutto: i checkpoint del lavoro

Con Claude Code si lavora per iterazioni veloci: si prova, si cambia, a volte si sbaglia. Git è il modo per avere checkpoint del proprio lavoro e poter tornare indietro quando una vibata va male.

Versionare

Salvare istantanee del progetto man mano che lavori. Se qualcosa si rompe, torni all'istantanea di prima.

È il "salva partita" del codice.

GitHub

Il posto nel cloud dove le istantanee vengono conservate: backup del progetto.

Il punto di partenza è banale: il progetto è una cartella sul vostro computer — file dentro una directory, niente di più. Git le aggiunge due cose: una memoria (ogni commit salva una nuova versione della cartella) e una copia nel cloud (il push la sincronizza su GitHub). Due gesti soli: niente branch, niente mondo infinito di Git.

02 Setup

Prima di poterlo usare: installare gli strumenti

Commit e push sono gesti che qualcuno deve saper eseguire sul vostro computer: quel qualcuno è git, più gh per parlare con GitHub. Si installano una volta sola, adesso, e non ci si torna più sopra. Questa è la procedura su Windows: si apre PowerShell — cercatelo nel menu Start — e da lì in poi si copia e incolla.

Sei passi · una volta sola

Installate Git e GitHub CLI. Due comandi, uno dopo l'altro. winget è l'installatore già presente in Windows: scarica e installa da solo.

PowerShell
winget install --id Git.Git -ewinget install --id GitHub.cli -e

Chiudete e riaprite PowerShell. Poi controllate che i due programmi rispondano: ognuno deve stampare il proprio numero di versione.

Non saltate questo passaggio. È il punto in cui si blocca quasi tutti: una finestra di PowerShell già aperta non vede i programmi appena installati. Se git o gh vi rispondono non riconosciuto, quasi sempre è solo questo — chiudete la finestra e riapritela.
PowerShell · finestra nuova
git --versiongh --version

Dite a Git chi siete. Nome ed email finiscono dentro ogni commit: è la firma delle vostre istantanee. Sostituite i valori con i vostri (l'email è quella dell'account GitHub). La terza riga dice a Git di chiamare main il ramo principale — è la convenzione di GitHub.

PowerShell
git config --global user.name "Nome Cognome"git config --global user.email "tua@email.it"git config --global init.defaultBranch main

Accedete a GitHub. Il comando fa quattro domande: rispondete come qui sotto muovendovi con le frecce e confermando con Invio. Alla fine si apre il browser: incollate il codice che vedete a schermo e autorizzate.

PowerShell
gh auth login

How would you like to authenticate?Login with a web browser

Collegate Git a GitHub. Questo passaggio consegna a Git le credenziali appena ottenute: da qui in avanti git push non vi chiederà mai una password.

PowerShell
gh auth setup-git

Verificate. Il primo comando deve dirvi che siete loggati. Il secondo deve stampare una riga: è la prova che le credenziali sono state consegnate a Git. Se non stampa nulla, il passo 5 non ha attecchito — rifatelo.

PowerShell
gh auth statusgit config --global --get-regexp credential

Fatto questo, PowerShell potete anche chiuderlo: da qui in avanti non vi servirà più. Il lavoro vero — creare la repo, agganciarla, costruire — si fa parlando con Claude, come vedrete al capitolo 06.

03 Perché un'app

Oggi non chiediamo un artifact: costruiamo un'app

Ieri con Cowork e gli artifact avete già costruito piccoli strumenti — la matrice, un singolo file HTML autocontenuto. Oggi l'obiettivo è un gestionale per la manutenzione dei treni: dati che devono restare e aggiornarsi, regole che girano da sole, un ERP da interrogare, l'AI dentro. Un singolo file non basta più.

Artifact

un file, un foglio — e finisce lì

Una mini-app in un file HTML: interfaccia, logica e dati vivono tutti lì dentro. Perfetto per la matrice di ieri.

Ma i dati non sopravvivono e non si condividono, non si aggancia l'ERP, non ci lavorano più persone. Il tetto si tocca in fretta.

App

dati che restano, servizi che girano, l'AI a chiamata

Un progetto vero: dati che restano, un'interfaccia per gli utenti, regole e calcoli che girano da soli, collegamenti ad altri sistemi, l'AI chiamata via API.

È la forma dei tool interni che da oggi potete costruirvi.

04 L'architettura, già decisa

Un'app è fatta di pezzi: questi cinque

Sceglierli — quale database, quale linguaggio, come si parlano — è l'unico punto dove servirebbe davvero un ingegnere. Quel lavoro è già fatto: le scelte sono fissate una volta e finiscono in un file di testo che Claude Code legge da solo e a cui si attiene. Quel file lo vediamo nascere fra due capitoli. Qui basta sapere cosa fa ogni pezzo, per seguire cosa succede dove.

06 Il metodo

Un progetto non comincia con un prompt: comincia con una cartella. La si apre, e da lì in poi si parla — la repository, il file delle regole, il primo pezzo di software: tutto si chiede a Claude in italiano, dentro la stessa finestra. Gli strumenti del capitolo 02 sono già installati e da qui in avanti lavorano per voi, sotto il cofano. Qui sotto i cinque step si vedono su un progetto piccolo di esempio, diverso da quello della demo.

Si apre la cartella di lavoro — con dentro i documenti. Claude Code non è un posto dove si caricano dei file: vive dentro una cartella del vostro computer e vede solo quella. Sceglierla è il primo gesto — e prima di parlargli ci si copiano dentro i documenti che contano: il foglio di specifica, l'Excel con i dati veri, tutto ciò su cui il progetto si basa. Quello che non è nella cartella, per Claude non esiste.

Si chiede la repository — a parole. Il contenitore nel cloud del capitolo 01, quello verso cui farete push. Non serve andare su GitHub a crearlo a mano: lo si chiede in italiano, e Claude lo crea e lo aggancia alla cartella. Da qui i vostri checkpoint hanno dove andare.

Il comando /init: si detta il software e come si vuole lavorare. Il file delle regole non si scrive a mano. A /init si racconta cosa deve diventare il progetto e con quali vincoli — e lui, dopo aver letto i documenti che avete messo nella cartella, scrive il CLAUDE.md, che rileggerà da solo a ogni avvio successivo. Per questo i file vanno messi lì prima: /init guarda la cartella com'è in quel momento.

Da qui si inizia a lavorare. Solo consegne, in italiano, una per volta. L'architettura non va più ripetuta: è nel file.

A ogni pezzo che funziona, il checkpoint. Si torna al ciclo del capitolo 01 — commit e push — ma non si digita nulla: si chiede. I gesti sono quelli, la mano è di Claude.

📄 CLAUDE.md il risultato dello step 03, sul progetto degli ordini — Claude Code lo rilegge a ogni avvio 
# CLAUDE.md — Gestionale ordini

Costruisci questo progetto per una persona che non programma. Le scelte qui sotto sono già fatte: non cambiarle. Se una richiesta ti costringe a uscirne, fermati e spiegami il problema.

## Cosa fa

Gli ordini dei clienti arrivano per mail, scritti a mano libera. L'app li raccoglie, l'AI li trasforma in ordini ordinati (cliente, articoli, quantità, data) e una pagina ne mostra lo stato fino alla consegna.

## Le scelte (fisse)

| Pezzo | A cosa serve |
|---|---|
| Dove stanno i dati | un archivio ordinato, in una scatola isolata sul computer |
| Il motore che elabora | riceve le mail, applica le regole, chiama l'AI |
| Le pagine che si vedono | il sito che apri per guardare gli ordini |
| AI | si chiama **solo** dal motore; la chiave sta nel file `.env` |

## Regole

- **Semplice.** Nessun pezzo in più: se ne serve uno, chiedimelo prima.
- **Segreti.** Chiavi e password solo nel `.env`. Mai nel codice, mai su GitHub.
- **Si avvia in tre comandi.** Scritti nel README, sempre aggiornati.
- **Checkpoint.** A ogni pezzo che funziona: commit e push, messaggi brevi in italiano.
- **Parlami da non tecnico.** A fine lavoro, dimmi in una riga cosa hai toccato.

Questo è il file che /init vi restituisce: la richiesta del capo diventata scelte fisse, più le regole di lavoro. L'esempio è sul progetto degli ordini; per la demo e per l'esercitazione di oggi il CLAUDE.md — stesso schema, sul gestionale dei treni — ce l'avete già nella cartella.

07 Da vedere

Demo: la dashboard dei treni

Un gestionale per la manutenzione ordinaria dei mezzi, costruito dal vivo: dallo schema di prima all'app funzionante. Ogni passo usa il vocabolario appena introdotto.

🧑‍💼
La consegna
Noi lavoriamo nella mobilità: gestiamo una flotta di treni, e ogni mezzo ha delle revisioni obbligatorie da fare a scadenze fisse. Non è manutenzione a guasto: è manutenzione ordinaria programmata, cioè controlli che vanno fatti a prescindere, a scadenza fissa — a chilometraggio percorso o a tempo, dipende dal tipo di controllo. Dal nostro gestionale, comunque, la scadenza esce già calcolata come data: a voi serve solo quella. Qualche esempio di cosa intendo. C'è la verifica dell'usura di cerchioni e sale montate — le ruote del treno — cioè un controllo dimensionale per capire quanto sono consumate. C'è il controllo dell'impianto frenante: pastiglie, dischi, pressione dell'aria compressa. C'è l'ispezione del pantografo, il braccio che prende corrente dalla linea aerea. C'è il controllo del sistema di sicurezza SCMT, che però richiede un tecnico abilitato: quello non lo può fare chiunque. Sui mezzi diesel ci sono il motore e la trasmissione. E poi la climatizzazione e le porte automatiche. Quindi noi sappiamo già cosa va controllato — lo scope è noto — quello che non sappiamo è se salterà fuori un problema o no. E ogni voce ha la sua scadenza: non scadono tutte insieme, per questo serve tenerne traccia mezzo per mezzo. Oggi questa roba la teniamo un po' a mente noi, o su un foglio Excel. Vorrei una schermata unica che mi faccia vedere la situazione e mi aiuti a organizzare il lavoro dei manutentori. Cosa deve mostrarmi questa schermata principale. Voglio vedere i miei mezzi raggruppati per categoria — che ne so, "otto mezzi di categoria 3" — e per ciascun gruppo voglio capire subito quanti sono in regola e quanti no: un mezzo è da sistemare se ha almeno una revisione scaduta o in scadenza, non mi importa quale. Non mi serve il dettaglio tecnico di ogni singola voce di controllo: dev'essere una vista d'insieme, un colpo d'occhio. Le revisioni già scadute, però, voglio vederle subito e distinguerle da quelle che scadono a breve — quelle scadute sono un problema. Poi su questa schermata mi serve un bottone, "Organizza revisione", che quando lo premo mi prepara l'agenda dei manutentori: chi va dove, la mattina e il pomeriggio, in modo da incastrare bene i giri in base a chi è disponibile. Tenete conto che ogni manutentore è assegnato a un deposito e ha le sue specializzazioni — l'SCMT, il diesel — quindi non tutti possono fare tutto. Non deve essere chissà quale algoritmo di ottimizzazione, i mezzi non sono migliaia: basta che il giro venga fuori organizzato in modo sensato, mattina e pomeriggio. E poi, cosa importante: quando ho organizzato l'agenda, non deve partire tutto in automatico. Prima l'agenda me la fa vedere, io la controllo, e solo quando confermo io parte l'invio delle email ai manutentori — ognuno riceve la sua mail, con scritto il proprio giro della giornata: "oggi devi fare questo, in questo ordine". Alla mail va allegata anche la scheda del mezzo, la documentazione che il manutentore si porta dietro. Ultima cosa, da dove arrivano i dati. Internamente esiste l'ERP aziendale, da cui prendere lo stato dei mezzi e le scadenze delle revisioni. Per ora possiamo anche solo simulare questa fonte dati — l'importante è che il resto dell'app la chiami come se fosse vera, così poi la colleghiamo a quella reale in un secondo momento.

Questo è tutto quello che il cliente sa dirci. È il primo prompt della demo: si incolla così com'è, nella cartella dove /init ha già scritto il CLAUDE.md del gestionale treni.

Cosa c'è dentro questo file

È l'estrazione del 13 luglio 2026 dall'ERP di Movia S.p.A., un'azienda di trasporto ferroviario regionale. Cinque elenchi, uno dentro l'altro come le pagine di un raccoglitore.

I depositi — tre: Bologna San Donato, Ferrara Interporto, Rimessa Ravenna Scalo. Di ciascuno si sa quanti binari di manutenzione ha e se ha la fossa di ispezione, cioè lo scavo sotto il binario che serve per lavorare sotto al treno. Ravenna non ce l'ha: le cose "sotto cassa" lì non si possono fare.

I mezzi — quattordici, dagli elettrotreni nuovi alle automotrici diesel storiche. Ognuno ha matricola, modello, chilometri percorsi, deposito di appartenenza, linea su cui gira, e due campi che contano per la dashboard: la categoria (1, 2 o 3 — è il raggruppamento che il cliente chiede: otto mezzi sono di categoria 3) e lo stato (in servizio, fermo programmato, accantonato).

I manutentori — sei, con nome, email, deposito, specializzazioni (chi fa il diesel, chi è abilitato all'SCMT) e la disponibilità: per ogni giorno della settimana, la fascia in cui lavorano — 06:00-14:00 o 14:00-22:00. Sono la mattina e il pomeriggio su cui si costruisce l'agenda.

Le revisioni — ventidue, il cuore del file. Ognuna lega un mezzo a un tipo di controllo (la revisione ordinaria, oppure un controllo obbligatorio: freni, climatizzazione, porte, SCMT, motore diesel) e porta con sé la scadenza, lo stato (fatta o da_fare) e, se è già stata fatta, chi l'ha eseguita, quando e com'è andata. Otto sono fatte, quattordici da fare — e due sono già scadute, tutt'e due sull'ETR-350-04.

Le schede treno — una per mezzo: la lista dei controlli previsti, i documenti da portare a bordo e le avvertenze aperte. È l'allegato che finisce nella mail al manutentore.

Da guardare con attenzione: ETR-350-04 ha due controlli scaduti da giugno (il fermo è stato rinviato) ed è il mezzo messo peggio. ALn-668-3145, invece, non ha nessuna revisione aperta — ma è accantonato, cioè fuori servizio: contarlo tra i mezzi "in regola" sarebbe una lettura ingenua del dato.

08 Da fare

Hands-on: il documento del revisore

Esercitazione su consegna assegnata. Non un'app nuova: una pagina in più della stessa app appena vista — così si riusa tutto il contesto della demo. E niente scelte tecniche neanche qui: il CLAUDE.md del progetto è già nella cartella, a voi serve solo descrivere bene la consegna.

Hands-on · ~2 ore
🧑‍💼
La consegna
Restiamo sulla stessa app di stamattina — vi chiedo una pagina in più. Quando un revisore torna dal giro sui mezzi, mi consegna i suoi appunti: un foglio scritto a mano, un po' alla rinfusa. «Sull'ETR-350-04 le pastiglie sono quasi da cambiare, il pantografo è a posto; il diesel del 07 perde un filo d'olio.» Roba utile, ma così com'è, buttata lì, non ci faccio granché. Vorrei una pagina dove carico quel documento — lo trascino e basta — e l'app me lo trasforma in una lista di interventi ordinata: un intervento per riga, agganciato al mezzo giusto e alla revisione a cui si riferisce. I dati dei mezzi e delle revisioni sono già quelli della dashboard di prima, quindi il collegamento lo potete fare da lì. A mettere ordine negli appunti ci pensa l'intelligenza artificiale, dentro l'app: prende il testo com'è e ne ricava la lista. Ma poi quella lista voglio poterla rivedere io prima di salvarla — magari l'AI si perde una riga, o aggancia un intervento al mezzo sbagliato — e solo quando confermo diventa definitiva, pronta per chi la prenderà in carico.

Gli step

La pagina di caricamento. Una box in cui trascinare il documento del revisore (fornito già pronto insieme alla consegna).

Il collegamento. L'app aggancia il documento alle revisioni che quel revisore doveva fare — i dati sono già nel database della demo.

La chiave dell'AI. L'estrazione usa un modello a pagamento, e per chiamarlo serve una chiave. Copiatela dal documento condiviso e datela a Claude chiedendogli di custodirla in modo sicuro dentro la cartella di lavoro — nel file .env, che non finisce su GitHub. Mai incollarla nel codice.

L'estrazione. Dal testo destrutturato l'app estrae la lista degli interventi: qui entra l'AI dentro il software, che usa la chiave appena messa al sicuro.

Modifica e conferma. Gli interventi estratti si possono correggere e integrare; poi OK, e vengono salvati — qualcun altro li prenderà in carico.
