01 Git essenziale

Prima di tutto: i checkpoint del lavoro

Con Claude Code si lavora per iterazioni veloci: si prova, si cambia, a volte si sbaglia. Git è il modo per avere checkpoint del proprio lavoro e poter tornare indietro quando una vibata va male — e, come vedrete nel pomeriggio, è ciò che abilita Claude Code in cloud.

Versionare

Salvare istantanee del progetto man mano che lavori. Se qualcosa si rompe, torni all'istantanea di prima.

È il "salva partita" del codice.

GitHub

Il posto nel cloud dove le istantanee vengono conservate: backup del progetto e — dal pomeriggio — la porta d'ingresso di Claude Code in cloud.

Il punto di partenza è banale: il progetto è una cartella sul vostro computer — file dentro una directory, niente di più. Git le aggiunge due cose: una memoria (ogni commit salva una nuova versione della cartella) e una copia nel cloud (il push la sincronizza su GitHub). Due gesti soli: niente branch, niente mondo infinito di Git.

04 Da vedere

Demo: la dashboard dei treni

Un gestionale per la manutenzione ordinaria dei mezzi, costruito dal vivo: dallo schema di prima all'app funzionante. Ogni passo usa il vocabolario appena introdotto.

Demo · dashboard treni

La fonte dati: un ERP mockato. Un componente preparato in anticipo che, ricevute le chiamate, risponde con dati presi da un JSON prefatto — mezzi, manutentori con le loro disponibilità, revisioni con le scadenze. "Mockare" è a sua volta un concetto da fissare: per costruire l'app non serve il sistema vero.

La dashboard. Visione d'insieme dei mezzi con lo stato delle revisioni e dei controlli obbligatori a scadenza: otto mezzi di categoria tre, tre con revisione fatta, cinque da fare.

Il bottone "Organizza revisione". Crea l'agenda del giorno o della settimana dei manutentori, incrociando revisioni da fare e disponibilità.

Human in the loop. L'agenda proposta si rivede: solo quando l'utente conferma, parte la mail ai manutentori con il programma ("il tuo tragitto del giorno è questo").

L'AI dentro il software. La composizione della mail la scrive un modello, chiamato via API dall'app stessa. La mail allega anche la scheda treno del mezzo, la documentazione da portare con sé.

05 Da fare

Hands-on: il documento del revisore

Esercitazione su consegna assegnata. Non un'app nuova: una pagina in più della stessa app appena vista — così si riusa tutto il contesto della demo.

Hands-on · ~2 ore

La consegna

Il revisore, dopo il giro sui mezzi, produce un documento destrutturato: appunti sparsi su treni visti e interventi da fare. Costruite la pagina in cui quel documento si carica e diventa una lista di interventi strutturata, collegata alle revisioni di quel revisore, pronta per chi li prenderà in carico.

Gli step

La pagina di caricamento. Una box in cui trascinare il documento del revisore (fornito già pronto insieme alla consegna).

Il collegamento. L'app aggancia il documento alle revisioni che quel revisore doveva fare — i dati sono già nel database della demo.

L'estrazione. Dal testo destrutturato l'app estrae la lista degli interventi: qui entra l'AI dentro il software, con la chiave API distribuita in aula.

Modifica e conferma. Gli interventi estratti si possono correggere e integrare; poi OK, e vengono salvati — qualcun altro li prenderà in carico.
